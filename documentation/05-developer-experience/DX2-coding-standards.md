## 2. Java Style Rules

### Typing and design

```java
// ✅ Good — explicit types; use records for immutable value objects
public record ConfirmBookingCommand(
    UUID tenantId,
    UUID appointmentId,
    String correlationId
) {}

// ✅ Good — explicit return type on service methods
public AppointmentConfirmedEvent confirmBooking(ConfirmBookingCommand command) { ... }

// ❌ Bad — raw Object type at domain boundaries
public Object confirmBooking(Object command) { ... }
```

- Use Java records for commands, events, and DTOs (immutable by default)
- Use POJOs annotated with Lombok `@Value` or `@Data` only when records are not applicable
- No raw `Object` types at domain boundaries — use explicit types or generics
- Avoid `null` returns from service methods — use `Optional<T>` for nullable lookups
- No suppression of warnings (`@SuppressWarnings`) without a comment explaining why

### Naming conventions

| Element | Convention | Example |
|---|---|---|
| Classes, records, enums, interfaces | PascalCase | `BookingCommand`, `TenantRole` |
| Variables, methods, parameters | camelCase | `tenantId`, `confirmBooking` |
| Constants (static final) | SCREAMING_SNAKE_CASE | `MAX_BOOKING_ATTEMPTS` |
| Files | Match class name | `BookingCommandService.java` |
| Packages | lowercase, domain-scoped | `com.anjischedulo.booking.command` |
| Database columns | snake_case (mapped via `@Column(name=...)`) | `tenant_id` |
| Event names | `noun.verb` | `appointment.confirmed` |

## 3. Domain Model Conventions

### Entity IDs

All entity IDs are UUIDs generated by PostgreSQL (`gen_random_uuid()`). Never generate IDs in application code.

```java
// ✅ Generated by DB — read from result via RETURNING clause
UUID appointmentId = jdbcTemplate.queryForObject(
    "INSERT INTO appointments (...) VALUES (...) RETURNING appointment_id",
    UUID.class);

// ❌ Don't generate IDs in application code
UUID id = UUID.randomUUID();
```

### Tenant ID propagation

Every service method that touches tenant-scoped data must accept `tenantId` as an explicit parameter — never read it from a static context.

```java
// ✅ Explicit tenantId parameter
public List<Slot> getAvailableSlots(UUID tenantId, UUID staffId, LocalDate date) { ... }

// ❌ Reading tenant from static/global context
public List<Slot> getAvailableSlots(UUID staffId, LocalDate date) {
    UUID tenantId = TenantContextHolder.get(); // hidden dependency
}
```

### Business rule enforcement

Business rules (BR001–BR014) are enforced in the domain service layer, not in controllers or repositories. If a business rule needs to be checked, it belongs in the service.

```java
// ✅ BR003 enforced in service layer
public AppointmentId createBooking(CreateBookingCommand command) {
    Tenant tenant = tenantRepository.findById(command.tenantId())
        .orElseThrow(() -> new TenantNotFoundException(command.tenantId()));
    if (tenant.monthlyBookingCount() >= tenant.plan().monthlyBookingLimit()) {
        throw new BookingLimitExceededException("BR003");
    }
    // ...
}
```

## 4. Event Envelope Standard

Every domain event must include the mandatory envelope fields defined in `specs/ai/events.json`. Use the shared `EventEnvelope` record:

```java
// packages/shared-types/src/main/java/.../EventEnvelope.java
public record EventEnvelope<T>(
    UUID eventId,
    String eventType,
    String schemaVersion,
    UUID tenantId,
    String correlationId,
    String causationId,
    Instant occurredAt,
    String producer,
    T payload
) {}

// Usage in booking-command-service:
var event = new EventEnvelope<>(
    result.eventId(),             // UUID from DB
    "appointment.confirmed",
    "1.0",
    command.tenantId(),
    command.correlationId(),
    command.commandId(),
    Instant.now(),
    "booking-command-service",
    new AppointmentConfirmedPayload(...)
);
```

Never publish an event without all envelope fields. The outbox relay validates the envelope before publishing to SNS.

## 5. Error Handling

### Domain errors

Create typed exception classes for domain errors — never throw raw `RuntimeException` with a string message at domain boundaries.

```java
// ✅ Typed domain exception
public class SlotUnavailableException extends DomainException {
    public SlotUnavailableException(UUID slotId) {
        super("Slot " + slotId + " is no longer available", "SLOT_UNAVAILABLE", 409);
    }
}

// ❌ Raw exception
throw new RuntimeException("slot not available");
```

### Error response shape

All HTTP error responses follow the RFC 7807 problem detail format:

```json
{
  "type": "https://errors.anji-schedulo.com/slot-unavailable",
  "title": "Slot Unavailable",
  "status": 409,
  "detail": "The selected slot is no longer available. Please choose another time.",
  "correlation_id": "corr-xyz-789",
  "tenant_id": "tid-abc"
}
```

### No swallowed errors

```java
// ✅ Log and re-throw or handle explicitly
try {
    stripe.confirm(paymentIntentId);
} catch (StripeException e) {
    log.error("Stripe payment confirm failed: paymentIntentId={}", paymentIntentId, e);
    throw new PaymentFailedException(paymentIntentId);
}

// ❌ Silent swallow
try {
    stripe.confirm(paymentIntentId);
} catch (Exception ignored) { }
```

## 6. Logging

Use SLF4J with Logback (structured JSON output via `logstash-logback-encoder`). Never use `System.out.println`.

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.stereotype.Service;

@Service
public class BookingService {
    private static final Logger log = LoggerFactory.getLogger(BookingService.class);

    public AppointmentId confirmBooking(ConfirmBookingCommand command) {
        log.info("Booking confirmation started: appointmentId={} tenantId={}",
            command.appointmentId(), command.tenantId());
    }
}
```

Use SLF4J parameterised logging (`{}` placeholders) — never string concatenation in log calls (prevents toString() cost when logging is disabled).

**Never log PII.** Log IDs (`customer_id`, `appointment_id`), not names, emails, or phone numbers.

## 7. Testing Requirements

Every PR must maintain or improve test coverage:

| Test type | Coverage target | Framework |
|---|---|---|
| Unit tests | 80% line coverage per service | JUnit 5 + Mockito |
| Integration tests | All saga flows covered | JUnit 5 + Testcontainers |
| Cross-tenant isolation | Mandatory — all tenant-scoped APIs | JUnit 5 + Testcontainers |

### Unit test structure

```java
class BookingServiceTest {

    @Test
    void confirmsBookingWhenSlotIsAvailableAndPaymentSucceeds() { ... }

    @Test
    void throwsSlotUnavailableExceptionWhenSlotIsLocked() { ... }

    @Test
    void throwsBookingLimitExceededExceptionWhenTenantAtMonthlyLimit_BR003() { ... }

    @Test
    void compensatesSlotReleaseWhenPaymentFails_BR004() { ... }
}
```

Test names describe the behaviour and reference the business rule suffix when relevant.

### No mocking the database

Integration tests use a real PostgreSQL instance (Docker Compose). Mocking the database is not permitted — prior incidents showed mocked DB tests passing while real DB migrations broke prod.

## 8. Code Organisation

### Service structure

```
services/{service-name}/
  src/main/java/com/anjischedulo/{domain}/
    {Domain}Controller.java       (HTTP layer only — no business logic; @RestController)
    {Domain}Service.java          (business logic + BR enforcement; @Service)
    {Domain}Repository.java       (SQL queries via JdbcTemplate/JPA; no business logic)
    command/
      Create{Domain}Command.java  (Java record)
    event/
      {EventName}Event.java       (Java record)
    exception/
      {ExceptionName}Exception.java
    dto/
      {Domain}Response.java       (Java record)
  src/main/java/com/anjischedulo/shared/
    TenantContextFilter.java      (ThreadLocal tenant_id; sets SET LOCAL on connection)
    CorrelationIdFilter.java
  src/test/java/com/anjischedulo/{domain}/
    unit/
    integration/
```

### No business logic in controllers

Controllers validate input (class-validator DTOs) and call the service. All business logic lives in the service.

### No SQL in services

SQL belongs in the repository. Services call repository methods — never `this.db.query(...)` directly.

## 9. Static Analysis and Formatting

Enforced by CI — PRs with violations are blocked.

```bash
# Run Checkstyle (style rules)
./gradlew checkstyleMain checkstyleTest

# Run SpotBugs (bug pattern detection)
./gradlew spotbugsMain spotbugsTest

./gradlew spotlessApply
```

Configuration:
- Checkstyle: `config/checkstyle/checkstyle.xml` at repo root (Google Java Style)
- SpotBugs: `config/spotbugs/exclude.xml` for known false-positive suppressions
- Spotless: enforces Google Java Format (4-space indent, 120-char line width)
- No unused imports, no unused variables (enforced by Checkstyle)

## 10. Security Rules

- Never log secrets, API keys, JWTs, or PII (email, phone, name)
- Never store secrets in property files committed to git; use `@Value("${secret.path}")` bound from environment variables injected by ECS at startup
- Never call external APIs directly from repositories — only from services via the appropriate client
- All user input validated with Bean Validation (`@Valid`, `@NotNull`, `@Size`) on controller method parameters before reaching the service layer
- SQL queries use parameterised statements only (JdbcTemplate `?` placeholders or JPA named parameters) — never string concatenation
- `app.tenant_id` must be set via `SET LOCAL` at the start of every database transaction that touches tenant-scoped tables
