## 2. TypeScript Rules

### Strict typing

```typescript
// ✅ Good — explicit types at module boundaries
export async function confirmBooking(
  command: ConfirmBookingCommand,
): Promise<AppointmentConfirmedEvent> { ... }

// ❌ Bad — implicit any leaks into the codebase
export async function confirmBooking(command: any) { ... }
```

- No `any` — use `unknown` and narrow explicitly if the type is genuinely unknown
- No non-null assertion (`!`) without a comment explaining why the value cannot be null
- No `@ts-ignore` — fix the type error or use `@ts-expect-error` with a reason comment
- Return types must be explicit on all exported functions and class methods

### Naming conventions

| Element | Convention | Example |
|---|---|---|
| Classes, interfaces, types, enums | PascalCase | `BookingCommand`, `TenantRole` |
| Variables, functions, parameters | camelCase | `tenantId`, `confirmBooking` |
| Constants (module-level) | SCREAMING_SNAKE_CASE | `MAX_BOOKING_ATTEMPTS` |
| Files | kebab-case | `booking-command.service.ts` |
| NestJS modules | Feature folder | `apps/booking-command/src/booking/` |
| Database columns | snake_case (mapped in entity) | `tenant_id` |
| Event names | `noun.verb` | `appointment.confirmed` |

## 3. Domain Model Conventions

### Entity IDs

All entity IDs are UUIDs generated by PostgreSQL (`gen_random_uuid()`). Never generate IDs in application code.

```typescript
// ✅ Generated by DB — read from result
const appointment = await this.db.query(
  'INSERT INTO appointments (...) VALUES (...) RETURNING appointment_id',
);

// ❌ Don't generate IDs in application code
const id = uuid();
```

### Tenant ID propagation

Every service method that touches tenant-scoped data must accept `tenantId` as an explicit parameter — never read it from a global or thread-local.

```typescript
// ✅ Explicit tenantId parameter
async getAvailableSlots(tenantId: string, staffId: string, date: Date) { ... }

// ❌ Reading tenant from global context
async getAvailableSlots(staffId: string, date: Date) {
  const tenantId = RequestContext.current.tenantId; // hidden dependency
}
```

### Business rule enforcement

Business rules (BR001–BR014) are enforced in the domain service layer, not in controllers or repositories. If a business rule needs to be checked, it belongs in the service.

```typescript
// ✅ BR003 enforced in service layer
async createBooking(command: CreateBookingCommand) {
  const tenant = await this.tenantRepo.findById(command.tenantId);
  if (tenant.monthlyBookingCount >= tenant.plan.monthlyBookingLimit) {
    throw new BookingLimitExceededException(BR003);
  }
  ...
}
```

## 4. Event Envelope Standard

Every domain event must include the mandatory envelope fields defined in `specs/ai/events.json`. Use the shared `EventEnvelope` type:

```typescript
import { EventEnvelope } from '@anji-schedulo/shared-types';

const event: EventEnvelope<AppointmentConfirmedPayload> = {
  event_id: result.event_id,           // UUID from DB
  event_type: 'appointment.confirmed',
  schema_version: '1.0',
  tenant_id: command.tenantId,
  correlation_id: command.correlationId,
  causation_id: command.commandId,
  occurred_at: new Date().toISOString(),
  producer: 'booking-command-service',
  payload: { ... },
};
```

Never publish an event without all envelope fields. The outbox relay validates the envelope before publishing to SNS.

## 5. Error Handling

### Domain errors

Create typed error classes for domain errors — never throw raw `Error` with a string message at domain boundaries.

```typescript
// ✅ Typed domain error
export class SlotUnavailableException extends DomainException {
  constructor(slotId: string) {
    super(`Slot ${slotId} is no longer available`, 'SLOT_UNAVAILABLE', 409);
  }
}

// ❌ Raw error
throw new Error('slot not available');
```

### Error response shape

All HTTP error responses follow the RFC 7807 problem detail format:

```json
{
  "type": "https://errors.anji-schedulo.com/slot-unavailable",
  "title": "Slot Unavailable",
  "status": 409,
  "detail": "The selected slot is no longer available. Please choose another time.",
  "correlation_id": "corr-xyz-789",
  "tenant_id": "tid-abc"
}
```

### No swallowed errors

```typescript
// ✅ Log and re-throw or handle explicitly
try {
  await this.stripe.confirm(paymentIntentId);
} catch (err) {
  this.logger.error({ err, paymentIntentId }, 'Stripe payment confirm failed');
  throw new PaymentFailedException(paymentIntentId);
}

// ❌ Silent swallow
try {
  await this.stripe.confirm(paymentIntentId);
} catch { }
```

## 6. Logging

Use the shared `Logger` wrapper (pino-based). Never use `console.log`.

```typescript
import { Logger } from '@anji-schedulo/shared-logger';

@Injectable()
export class BookingService {
  private readonly logger = new Logger(BookingService.name);

  async confirmBooking(command: ConfirmBookingCommand) {
    this.logger.info(
      { appointmentId: command.appointmentId, tenantId: command.tenantId },
      'Booking confirmation started',
    );
  }
}
```

**Log objects, not strings.** Put structured data in the first argument (object), human text in the second.

**Never log PII.** Log IDs (`customer_id`, `appointment_id`), not names, emails, or phone numbers.

## 7. Testing Requirements

Every PR must maintain or improve test coverage:

| Test type | Coverage target | Framework |
|---|---|---|
| Unit tests | 80% line coverage per service | Jest |
| Integration tests | All saga flows covered | Jest + Docker Compose |
| Cross-tenant isolation | Mandatory — all tenant-scoped APIs | Jest integration |

### Unit test structure

```typescript
describe('BookingService', () => {
  describe('confirmBooking', () => {
    it('confirms booking when slot is available and payment succeeds', async () => { ... });
    it('throws SlotUnavailableException when slot is locked', async () => { ... });
    it('throws BookingLimitExceededException when tenant at monthly limit (BR003)', async () => { ... });
    it('compensates slot release when payment fails (BR004)', async () => { ... });
  });
});
```

Test names describe the behaviour and reference the business rule when relevant.

### No mocking the database

Integration tests use a real PostgreSQL instance (Docker Compose). Mocking the database is not permitted — prior incidents showed mocked DB tests passing while real DB migrations broke prod.

## 8. Code Organisation

### Service structure

```
apps/{service-name}/
  src/
    {domain}/
      {domain}.module.ts
      {domain}.controller.ts    (HTTP layer only — no business logic)
      {domain}.service.ts       (business logic + BR enforcement)
      {domain}.repository.ts    (SQL queries — no business logic)
      dto/
        create-{domain}.dto.ts
        {domain}-response.dto.ts
      events/
        {event-name}.event.ts
      exceptions/
        {exception-name}.exception.ts
    shared/
      tenant-context.middleware.ts
      correlation-id.middleware.ts
  test/
    unit/
    integration/
```

### No business logic in controllers

Controllers validate input (class-validator DTOs) and call the service. All business logic lives in the service.

### No SQL in services

SQL belongs in the repository. Services call repository methods — never `this.db.query(...)` directly.

## 10. Security Rules

- Never log secrets, API keys, JWTs, or PII (email, phone, name)
- Never store secrets in `.env` files committed to git
- Never call external APIs directly from repositories — only from services via the appropriate client
- All user input validated with class-validator DTOs before reaching the service layer
- SQL queries use parameterised statements only — never string concatenation
- `app.tenant_id` must be set via `SET LOCAL` at the start of every database transaction that touches tenant-scoped tables
